# Seeded SFS Core Documentation

This document explains the seeded sequential forward selection (Seeded SFS) implementation used in this repository, including runtime flow, configuration, outputs, and common pitfalls.

## 1) Purpose

Seeded SFS is a wrapper feature selection approach that starts from pre-ranked seed features (from ensemble voting) and then performs forward selection with cross-validated scoring.

Compared to sklearn SFS, this implementation adds:

- explicit seed initialization,
- global-best rollback behavior,
- iteration-level history and timing,
- integrated output artifact saving through the wrapper framework.

## 2) Architecture Map

### Core algorithm

- `src/wrapper/forward_selection.py`
  - `SeededForwardSelection`: custom sklearn-compatible selector (`fit`, `transform`, support mask, history/report).

### Wrapper orchestration

- `src/wrapper/seeded.py`
  - `SeededSFSSelector`: bridges data, seed file path, algorithm execution, and SFSResult creation.
- `src/wrapper/base.py`
  - `BaseWrapperSelector`: common `run_sfs(...)` flow and artifact persistence.

### Supporting utilities

- `src/utils/utils.py`
  - `load_seed_from_csv(...)`, `validate_features(...)`, `create_union_features(...)`.
- `src/wrapper/models.py`
  - model factory used by seeded selector (`logistic`, `rf`, `svm`, `xgboost`, `dt`).
- `src/utils/experiment_paths.py`
  - run directory builder for `results/.../run_YYYYMMDD_HHMMSS...`.
- `src/config.py`
  - canonical project paths (`ProjectPath`).

### Typical entry scripts

- Raw variant: `notebook/<dataset>/07_sfs-raw.py`
- Union variant: `notebook/<dataset>/07_sfs-union.py`

## 3) Input Contracts

## Dataset DataFrame

- First column must be target label `y`.
- Remaining columns are candidate features `X`.

`BaseWrapperSelector.run_sfs(...)` assumes this format and splits:

- `y_in = df.iloc[:, 0]`
- `X_in = df.iloc[:, 1:]`

## Seed voting CSV

- Usually located under `data/processed/<dataset>/03_ensemble/`.
- Required columns:
  - `Feature`
  - `Votes`
- Seed extraction uses top rows: `df["Feature"].head(n_seeds)`.

## Variant inputs

- `raw`: data loaded from `ProjectPath.raw_path`.
- `union`: data typically generated by `create_union_features(...)` from filter outputs, then passed to Seeded SFS.

## 4) End-to-End Runtime Flow

1. Notebook sets dataset config (`data_name`, `n_features`, `voting_csv_name`, SFS params).
2. Notebook creates `SeededSFSSelector(...)` with `dataset_variant` (`raw` or `union`).
3. `run_sfs(...)` in `src/wrapper/base.py`:
   - splits input dataframe into `X` and `y`,
   - builds `sfs_params`,
   - calls `_execute_core(...)`.
4. `_execute_core(...)` in `src/wrapper/seeded.py`:
   - resolves seed CSV path (`self.path.ensemble_dir / voting_csv_name`),
   - instantiates `SeededForwardSelection(...)`,
   - executes `fit(X, y)` then `transform(X)`,
   - builds final dataframe (`y + selected features`),
   - returns `SFSResult`.
5. `BaseWrapperSelector._save_sfs_output(...)` persists outputs:
   - processed wrapper csv,
   - history txt,
   - selected features csv,
   - metrics json/csv.

## 5) Core Algorithm Behavior (`SeededForwardSelection`)

## Initialization

- Resolve model:
  - if `model` is string -> `get_model(...)` factory.
  - if estimator instance -> use directly.
- Load seeds from:
  - list of feature names, or
  - CSV path via `load_seed_from_csv(...)`.
- Validate seeds exist in current `X` columns (`validate_features`).

## CV strategy

- Build CV splitter once via `_build_cv()`:
  - `StratifiedKFold` when `cv_stratified=True`, else `KFold`.
  - deterministic with `random_state` if `cv_shuffle=True`.
- Freeze splits in `_cv_splits_` and reuse in every scoring call.

## Baseline

- Evaluate score of initial seed set only.
- Initialize:
  - `current_score`,
  - `global_best_score`,
  - `global_best_indices`.

## Iterative forward selection

For each iteration:

1. Get candidate features not in current selected set.
2. Evaluate each candidate in parallel:
   - score(`selected + candidate`) with `cross_val_score(...)`.
3. Pick candidate with highest score for that iteration.
4. Append candidate to current selected set.
5. Update:
   - step improvement (`best_score - current_score`),
   - global best + patience counter,
   - history row with timing.

## Stop conditions

- No candidate features left.
- `max_features` reached.
- `patience` exhausted (no global improvement for N iterations).

## Finalization

- Persist final selected features from `global_best_indices` (best observed, not necessarily last iteration).
- Expose fitted attributes:
  - `selected_features_`, `global_best_score_`,
  - `history_`, `total_iter_time_ms_`, `iteration_time_ms_`,
  - `total_fit_time_ms_`.

## 6) Parameter Reference

### SeededForwardSelection defaults

- `n_seeds=1`
- `model="logistic"`
- `scoring="accuracy"`
- `cv=5`
- `cv_shuffle=True`
- `cv_stratified=True`
- `max_features=100`
- `patience=5`
- `random_state=42`
- `verbose=2`
- `n_jobs=-1` (candidate-level parallelism)
- `using_timer=True`
- `unit="ms"`

### Base wrapper `run_sfs(...)` defaults

- `max_features=20`
- `cv=5`
- `verbose=2`
- `model="logistic"`
- `scoring="accuracy"`
- `direction="forward"`

Additional kwargs (for example `n_seeds`, `patience`) are passed through to `SeededForwardSelection`.

## 9) Known Gotchas and Troubleshooting

- Seed CSV missing or malformed:
  - file not found -> `FileNotFoundError`
  - missing `Feature`/`Votes` -> `ValueError`
- Seed not present in dataset columns -> `ValueError` from `validate_features(...)`.
- `cv < 2` -> invalid; `_build_cv()` raises error.
- Stop checks happen after each iteration, so one iteration is always attempted before checking `max_features`/`patience`.
- Seed deduplication is not explicit; duplicated seed names can lead to duplicated starting indices.
- Legacy file `src/wrapper/old_SFS.py` exists; active implementation is `src/wrapper/forward_selection.py`.

## 10) Reproducibility and Performance Notes

- Reproducibility:
  - deterministic CV splits are frozen once per fit,
  - `random_state` controls model and shuffled CV behavior (where applicable).
- Performance:
  - candidate scoring uses `joblib.Parallel(n_jobs=self.n_jobs)`,
  - each internal `cross_val_score(...)` call uses `n_jobs=1` to reduce nested parallel oversubscription.
